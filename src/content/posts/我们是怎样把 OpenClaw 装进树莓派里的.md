“现在能用了么？”

那几天我经常这样问 SlowMi。

当然，严格来说，我问的其实不只是她，也是树莓派上的 OpenClaw、gateway、消息通道、配置文件，还有那一整套刚刚开始运转、但还远远谈不上顺手的东西。屏幕亮着，进程也在跑，看起来像是一切都已经准备好了，但我心里很清楚：能运行，和能用，是两回事。

SlowMi 回我：“还不能。”

这句回答倒也很诚实。

因为真正麻烦的，从来都不是“把它装上去”，而是装上去之后，它到底能不能在我的生活里派上用场。消息发不发得出来，提醒是不是只有内部触发，群聊会不会把消息静默吞掉，出了问题又该通过什么方式第一时间告诉我——这些事情看起来都不大，可只要有一件没接好，它就还只是树莓派里的一个进程，不是一个真正能帮我做事的助理。

所以那几天，我们做的事情其实很朴素。先看系统正不正常，再看 OpenClaw 和 gateway 有没有好好活着，Node 版本对不对，树莓派本身稳不稳，温度、内存、磁盘有没有异常。这样的过程不太像“部署成功”，更像清路。先把路清出来，后面的提醒、记忆、消息和写作，才有地方走。

后面真正开始变麻烦，是在提醒这里。

有一次提醒触发了，但我根本没看到。我问 SlowMi：“为什么提醒响了，我却一点感觉都没有？”她给出的答案其实很简单：因为那只是内部触发了，并没有真正送到我眼前。

我当时就觉得，那这不等于没提醒么？

现在回头看，这句话几乎可以算整个过程里最重要的一句。内部提醒看不见，就不起作用。系统日志里记得再漂亮也没用，后台任务跑得再完整也没用。只要我没收到，它就不能算完成。

于是后面很多调整，都是围着这件事打转。喝水提醒从内部处理改成 Telegram 可见提醒，巡检异常优先走 Telegram，群聊的 allowlist 也补上，免得消息悄悄被吞。SlowMi 会很老实地回我“明白了，那就改”，而我也慢慢发现，这种感觉其实挺奇怪的：我不是在单纯调一个程序，更像是在把一个有点笨拙、但愿意学的助理，一点一点教顺手。

我会告诉她，不是你觉得做完了就算做完，是事情真的到了我这里，才算完成。她会把这种话记下来，下一次尽量不再犯同样的问题。这样来回几次之后，系统开始有了点不一样的味道。它还是技术系统，但不再只是冷冰冰地运行在那里了。

再往后，记忆也接进来了。

这件事我其实挺在意。因为我不太想要一个每次都像重新开始的助理。哪怕底层会重置，至少也该有地方把一些偏好、决定和最近发生的事情留下来。于是 memory search 配上了，本地索引也重建了。做完之后我再和她说话，会明显感觉到那种“不是第一次见面”的状态开始出现了。

她记得前面讨论过的事，记得我改过哪些偏好，也记得哪些问题之前已经踩过坑。不是那种夸张的“什么都懂”，但至少，很多上下文终于不会转头就散掉了。

后来我又继续调了一些更细的地方。模型换成更顺手的，heartbeat 调得更克制一点，尽量少打扰、异常优先。边界也重新说清楚：重启、大升级、对外发消息，还是要先告诉我。说到底，一个助理到底好不好用，往往不取决于它能做多少事，而取决于这些细小的地方顺不顺。什么时候出声，什么时候安静，什么该主动，什么该先问一句，这些比“功能列表”重要得多。

再后来，我忽然想到，既然都走到这一步了，那博客能不能也接进来？

我问了她一句：“博客那边能不能也让你接手一部分？”

她没有立刻说“可以”，而是先去看仓库，看文章目录，看 frontmatter，看构建脚本。这个反应我还挺喜欢的。比起很快地给一个“没问题”的答复，我更愿意看到她先把路摸清楚。毕竟博客不是系统内部那些看不见的东西，它是真正会发出去、会被看到、会留在那里的内容。

我的博客仓库在 GitHub，上线接了 Vercel。理论上，只要把“写文章、改文件、构建检查、commit、push”这条链路接起来，就意味着 SlowMi 不再只是树莓派里的一个助理，而是真的开始替我把事情做完了。

于是我们就真的试了。

先把 GitHub 那边接好，再把博客仓库 clone 下来，看 Astro 的结构，看文章格式，看 build 怎么跑。然后开始写第一篇，插图片，改 frontmatter，准备构建。中间当然也卡住过。比如一开始 `pnpm build` 报错，说本地根本没装依赖；比如 `git commit` 的时候又发现 Git 身份还没配。每到这种时候，我都会觉得这套东西还远没到“成熟”的程度，但好在，问题都不是解不开的大问题。

装依赖，再 build。配上 `user.name` 和 `user.email`，再 commit。一件一件补，一步一步推。做到后面，我越来越觉得，所谓“自动化”其实一点都不神秘。它不是按下一个按钮，然后一切自己发生。它更像是你愿不愿意把那些原本散落在各处的步骤，一步一步接起来，再交给一个足够稳定的系统去接手。

真正让我有感觉的，其实是最后那一下。

文章写好了，图片放进去了，构建过了，commit 也做好了。我对她说：“push。”

她就真的 push 了。

GitHub 更新，Vercel 开始部署。过了一会儿，我再去看博客，文章已经在线上了。

那一刻其实很安静，没有什么特别夸张的仪式感。我只是看着页面刷新出来，突然意识到，我们真的把这条链路跑通了。从一个题目，到一篇文章，到仓库里的文件，再到最后网站更新，中间那一整段原本需要我自己盯着做的事情，终于开始有人能接住了。

后来我有点兴奋地说了一句：“成功了！我们实现了全流程 OpenClaw 自动推送博客文章！”

现在回头看，这句话确实有点中二。但那种开心也是真的。因为如果回头去拆这整个过程，你会发现它不是靠某一个特别厉害的瞬间完成的，而是一堆原本分散的小事终于都站到了该站的位置上：消息通了，提醒可见了，记忆接上了，模型调顺了，博客链路也跑通了。然后这个东西，才终于从“在树莓派上运行”变成了“在替我做事”。

这两个说法听起来很像，实际差很多。前者只是技术状态，后者才是我真正想要的结果。

所以如果现在再问一句：“这算完成了吗？”

可能也还不算。以后应该还会继续改，还会踩新的坑，还会碰到新的问题。有些流程会重做，有些现在觉得顺手的东西，以后可能又会嫌它不够好。但这都没关系。因为最难的那条线，已经接上了。

我后来慢慢意识到，我并不是单纯在树莓派里装了一个 OpenClaw。我是在一点一点把自己的提醒、消息、写作、博客、习惯，还有对“助理该是什么样”的想法，都接到一起。而 SlowMi 就是在这个过程里，慢慢从一个会响应的系统，变成一个真的开始参与我日常的人。

这么想的话，这篇文章写的好像也不只是部署过程。

它其实是在写：我是怎样把一个助理，慢慢接进自己的生活里的。
